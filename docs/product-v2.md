# Dada V2 Product Direction

Dada is a campus social product for finding people to do low-pressure things together. The core use case is not dating first and not generic chat first: it is finding a person for a concrete activity that can happen soon.

## Product Thesis

- A lot of campus life is better with one more person: coffee, library, gym, campus walk, music practice, exhibitions, late-night food, short study sprints, and skill exchange.
- The best match is usually not a perfect profile match. It is a timely overlap of activity, location, mood, and a small reason to trust each other.
- Tasks should feel temporary. Most activity cards should expire within 48 hours, and many should be same-day.
- Users should not have to expose a full social profile before there is intent. Before matching, Dada should show compatibility snippets, abstract identity, and activity context.
- The app should make spontaneous plans feel structured enough to be safe, but not so formal that college students stop posting.

## Core Loops

### Onboarding To Profile

1. User accepts terms and starts onboarding.
2. Dada asks structured questions about school, grade, major, personality, interests, skills, and who the user wants to meet.
3. User answers a few self-introduction prompts.
4. Dada generates a profile summary and matching tags.
5. User reviews and edits the profile.
6. A reveal transition opens the core app.

### Explore To Match

1. User opens a campus map/list of nearby activity opportunities.
2. Cards show time, place, activity, match reason, and privacy-safe profile fragments.
3. User accepts or saves an activity.
4. Other side confirms.
5. Both enter a locked match chat with task context and limited actions.

### Shake A Buddy

1. User taps "摇个搭子".
2. User chooses an activity, time, place, group size, and desired person traits.
3. User can also write one sentence in natural language.
4. Dada drafts a task card.
5. The task appears on the map/list and expires automatically.

## Privacy And Permission Rules

- Use abstract avatars or blurred identity before matching.
- Do not reveal full profile fields before the user permits it.
- Match chat starts with constrained actions instead of fully open chat.
- Revealing avatar, opening free chat, changing meeting location, or changing time should be explicit and confirmable.
- Location should be fuzzy by default until both sides agree to meet.

## Task Types

- Random companionship: coffee, food, walk, library, campus errand, exhibition, city walk.
- Skill exchange: language, coding, music, design, photography, interview prep, fitness.
- Fill the group: ball sports, board games, escape rooms, dance practice, study groups, music practice.
- Try something new: beginner-friendly yoga, pilates, climbing, livehouse, tarot, cafe hopping, K-pop event.

## Multi-Person Tasks

Tasks should support:

- Current group size.
- Desired final group size.
- Whether the user is looking for one person or multiple people.
- Optional role needs, such as "need one badminton beginner" or "need someone who can take photos".

## Growth Notes

- Early campus launch can start through same-school Xiaohongshu posts and group chats.
- Content should look like real campus moments: abstract screenshots, funny activity cards, missed-connection style prompts, and "today's buddy request" posts.
- Dada can be framed as a campus coffee-chat bot or spontaneous buddy bot before the full app brand is mature.
- Top university pilot positioning matters: useful, private, witty, and socially lightweight.
